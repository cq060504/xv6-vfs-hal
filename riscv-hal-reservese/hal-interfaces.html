统一入口 hal/hal.h

1. 作用

hal.h 是整个 HAL 层的唯一入口头文件。内核代码只需 #include "hal/hal.h" 即可获得全部 HAL 功能，无需关心底层是 RISC-V 还是 LoongArch。

2. 关联关系

// hal.h 的 include 链 hal/hal.h ├── types.h // 基础类型定义 ├── 根据 ARCH 宏选择平台: │ ├── ARCH_riscv → hal/riscv/arch.h // RISC-V CSR 内联函数 + PTE 宏 │ │ hal/riscv/memlayout.h // RISC-V 物理地址布局 │ └── ARCH_loongarch → hal/loongarch/arch.h / memlayout.h └── 7 个子系统 HAL 头文件: ├── hal_arch.h // CPU 控制 ├── hal_vm.h // 虚拟内存 ├── hal_intr.h // 中断控制器 ├── hal_timer.h // 定时器 ├── hal_memlayout.h // 内存布局常量 ├── hal_console.h // 控制台 I/O └── hal_ctx.h // 上下文切换

被引用者（19 个内核文件 + 4 个 HAL 实现文件均 include 它）：

kernel/main.c kernel/proc.c kernel/vm.c kernel/trap.c kernel/syscall.c kernel/sysproc.c kernel/sysfile.c kernel/exec.c kernel/fs.c kernel/bio.c kernel/log.c kernel/file.c kernel/pipe.c kernel/console.c kernel/printf.c kernel/kalloc.c kernel/sleeplock.c kernel/spinlock.c kernel/riscv.h —— 以及 hal/riscv/hal_start.c hal/riscv/hal_plic.c hal/riscv/hal_uart.c hal/riscv/hal_virtio.c

3. 特殊设计点

hal.h 本身的职责是"聚合"——它不声明任何函数，只做两件事：

  1. 通过 #ifdef ARCH_xxx 条件编译，选择对应平台的 arch.h 和 memlayout.h（这两个文件提供平台相关的内联函数和物理地址宏）。
  2. 依次引入 7 个子系统 HAL 头文件，让内核一次性拿到全部 HAL 声明。
关键设计决策：将平台头文件（arch.h、memlayout.h）放在子系统头文件（hal_arch.h、hal_memlayout.h）之前引入。因为子系统头文件中的一些声明依赖平台提供的类型和宏（如 uint64、PTE_V 等）。
CPU 控制 hal/hal_arch.h

1. 作用

定义与 CPU 特权架构相关的所有操作接口：读取/写入控制状态寄存器（CSR）、获取核心 ID、控制中断使能、读写异常状态等。这些是操作系统与 CPU 硬件交互的最底层原语。

2. 关联关系

平台实现：RISC-V 架构中，所有 CSR 操作通过 hal/riscv/arch.h（~367行）中的 static inline 函数实现，编译为单条 CSR 指令。LoongArch 需要提供自己的 hal/loongarch/arch.h。

主要调用者：

kernel/trap.c — 读取 scause/stval/sepc/sstatus，写 stvec/sepc/sstatus
kernel/vm.c — 写 satp，调用 sfence.vma
kernel/proc.c — 读取 tp (hartid)
kernel/spinlock.c — intr_on/intr_off（开关中断保护临界区）
hal/riscv/hal_start.c — M 态 CSR 初始化（mstatus/mepc/medeleg/mideleg 等）

3. 函数列表

函数声明功能RISC-V 实现状态
hal_get_hartid() 获取当前 CPU 核心号（hart ID），用于多核调度区分 CPU r_mhartid() → csrr mhartid 内联实现
hal_read_sstatus() 读 S 态状态寄存器（中断使能位 SIE、特权级 SPP 等） r_sstatus() → csrr sstatus 内联实现
hal_write_sstatus() 写 S 态状态寄存器，用于修改中断使能状态 w_sstatus() → csrw sstatus 内联实现
hal_read_sepc() 读 S 态异常程序计数器（触发异常的那条指令的地址） r_sepc() → csrr sepc 内联实现
hal_write_sepc() 写 S 态异常程序计数器（设置异常返回后的目标地址） w_sepc() → csrw sepc 内联实现
hal_read_scause() 读异常原因寄存器（判断是中断、系统调用、缺页还是非法指令） r_scause() → csrr scause 内联实现

中断控制器 hal/hal_intr.h

1. 作用

定义平台级中断控制器（PLIC / EIOINTC）的编程接口。外设（磁盘、串口）产生中断后，由中断控制器汇总并路由到 CPU 核心。HAL 提供初始化、中断认领、中断完成确认四个标准操作。

2. 关联关系

平台实现：RISC-V 版本在 hal/riscv/hal_plic.c（~55行），直接操作 PLIC 的 MMIO 寄存器。LoongArch 需要实现 EIOINTC 版本。

主要调用者：

kernel/main.c — CPU0 调用 plicinit() + plicinithart()；其他 CPU 调用 plicinithart()
kernel/trap.c — 中断处理中调用 plic_claim() 认领 IRQ，处理完后调用 plic_complete() 确认

3. 函数列表

函数声明功能RISC-V 实现实现文件
hal_irq_init() 初始化整个中断控制器：设置各中断源的优先级阈值，禁止所有中断 plicinit() hal_plic.c
hal_irq_hart_init() 每个 CPU 核心的"本地"初始化：使能该核心接收的中断，设置该核心的优先级阈值 plicinithart() hal_plic.c
hal_irq_claim() 当 CPU 收到外部中断时，读 PLIC 获取当前最高优先级的中断号（IRQ）。这是"认领"操作 plic_claim() hal_plic.c
hal_irq_complete() 中断处理完毕后，向 PLIC 写回中断号，表示该中断已处理完成，可接收下次中断 plic_complete() hal_plic.c
RISC-V vs LoongArch 差异：RISC-V 的 PLIC 通过 MMIO 内存映射寄存器操作（地址 0x0c000000 起），而 LoongArch 使用 EIOINTC（扩展 I/O 中断控制器），寄存器布局和初始化流程完全不同。HAL 接口掩盖了这些差异，内核只需调用四个函数。
定时器 hal/hal_timer.h

1. 作用

定义定时器中断的配置和读取接口。操作系统依赖定时器中断实现时间片轮转调度（每个进程运行一个时间片后被抢占）、sleep 超时等。

2. 关联关系

平台实现：RISC-V 使用 SBI 的 stimecmp CSR（S 态定时器比较寄存器）和 CLINT（核心本地中断器）配合。实现在 hal/riscv/hal_start.c: timerinit() 和 arch.h: r_time()。LoongArch 使用 TCFG/TVAL CSR。

主要调用者：

kernel/trap.c — 在 kernelvec 定时器中断处理中调用 w_stimecmp(r_time() + interval) 设置下次时钟中断
hal/riscv/hal_start.c — timerinit() 在 M 态启动时配置第一次定时器中断
kernel/proc.c — 调度器隐式依赖定时器中断驱动的 yield()

3. 函数列表

函数声明功能RISC-V 实现实现文件
hal_get_time() 读取当前 CPU 周期计数器（自启动以来的 ticks），用于计时和设置下次中断 r_time() → csrr time arch.h 内联
hal_set_timer(next) 设置下次定时器中断的触发时刻（绝对时间值）。到达此时间时 CPU 收到定时器中断 w_stimecmp(next) arch.h 内联
hal_timer_init() 初始化定时器：使能 M 态定时器中断，设置第一次中断。在 M 态初始化阶段调用 timerinit() hal_start.c
时钟中断完整流程：timerinit()（M 态）→ M 态收到中断 → 转发到 S 态 → kernelvec.S → kerneltrap() → w_stimecmp(r_time() + interval) 设置下一次 → yield() 让出 CPU。
虚拟内存 hal/hal_vm.h

1. 作用

定义虚拟内存管理和 MMU 控制的最小接口。xv6 的大部分页表操作（walk、mappages、uvmalloc 等）在 kernel/vm.c 中实现，因为页表遍历算法是架构无关的。真正需要 HAL 抽象的是 TLB 刷新指令和页表格式常量。

2. 关联关系

平台实现：RISC-V 的 TLB 刷新通过 arch.h: sfence_vma() 内联函数实现。PTE 格式常量（PTE_V、PTE_R、PTE_W、PTE_X、PTE_U）和页大小（PGSIZE=4096、PGSHIFT=12）也定义在 arch.h 中。

主要调用者：

kernel/vm.c — kvminithart() 中调用两次 sfence_vma()（切换页表前后各一次清除旧 TLB 条目）
kernel/proc.c — 进程页表操作使用 PTE_* 标志
所有映射内存的代码 — 使用 PGSIZE、PGSHIFT、PGROUNDUP/DOWN 宏

3. 函数列表

函数声明 / 常量功能RISC-V 实现实现文件
hal_tlb_flush_all() 刷新全部 TLB 条目。切换页表后必须调用，否则旧页表的缓存映射会导致错误 sfence_vma() → sfence.vma zero, zero arch.h 内联
PGSIZE 页大小（4096 字节） #define PGSIZE 4096 arch.h
PGSHIFT 页内偏移比特数（12） #define PGSHIFT 12 arch.h
PTE_V / R / W / X / U 页表项标志位：有效、可读、可写、可执行、用户可访问 #define PTE_V (1L << 0) … arch.h
MAXVA 最大虚拟地址（Sv39：2³⁸⁻¹） 1L << (9+9+9+12-1) arch.h
为什么 hal_vm.h 很"空"：页表 walk、mappages、uvmalloc 等核心逻辑在不同的页表格式（Sv39 / LoongArch PGDL）下算法完全相同——都是 3 级页表，9-9-9-12 划分。差异仅在于 PTE 标志位的具体编码和 TLB 刷新指令。因此 HAL 只需抽象这些差异点，无需重复页表算法。
内存布局 hal/hal_memlayout.h

1. 作用

声明物理内存布局中的架构无关常量和符号。具体的物理地址值（UART0、PLIC、KERNBASE 等）由各平台的 memlayout.h 提供。本文件只声明内核镜像边界符号（由链接脚本导出）。

2. 关联关系

平台实现：hal/riscv/memlayout.h（~60行）定义 RISC-V virt 机器的全部物理地址。架构无关的宏（如 KSTACK、TRAMPOLINE、TRAPFRAME）也在其中，因为它们的计算依赖平台相关常量。

主要调用者：

kernel/vm.c — kvminit() 中映射 UART0/PLIC/VIRTIO0/KERNBASE/TRAMPOLINE 等所有物理地址区域
kernel/proc.c — KSTACK(p) 计算每个进程的内核栈地址；TRAMPOLINE/TRAPFRAME 映射到用户页表
kernel/kalloc.c — 使用 PHYSTOP 确定物理内存上限
hal/riscv/hal_uart.c — 使用 UART0 基地址
hal/riscv/hal_plic.c — 使用 PLIC 基地址和子寄存器偏移

控制台 I/O hal/hal_console.h

1. 作用

定义控制台输入输出接口。xv6 通过 UART 串口与外界通信（键盘输入、屏幕输出）。HAL 将串口硬件操作抽象为字符级 I/O 接口。同时提供中断驱动的异步输出路径。

2. 关联关系

平台实现：RISC-V 版本在 hal/riscv/hal_uart.c（~150行），操作 16550a UART 的 MMIO 寄存器。内核的 kernel/console.c 是 UART 驱动的"消费者"——它调用 uart 函数，并提供行缓冲、回显、Ctrl 组合键处理等高层功能。

调用链：

kernel/main.c → consoleinit() → uartinit()
kernel/printf.c → consputc() → uartputc_sync()（同步输出）/ uartputc()（中断驱动输出）
kernel/console.c: consoleintr() → 被 uartintr() 在每次收到字符时回调
kernel/trap.c → PLIC 认领 UART0_IRQ(10) → uartintr() → consoleintr()

3. 函数列表

函数声明功能RISC-V 实现实现文件
hal_console_init() 初始化串口硬件：设置波特率、数据位、停止位、使能 FIFO、配置中断 uartinit() hal_uart.c
hal_putchar_sync(c) 同步输出一个字符。等待发送寄存器为空后写入。用于内核启动早期（中断尚未就绪）和 printf 输出 uartputc_sync(c) hal_uart.c
hal_putchar(c) 异步输出一个字符。将字符放入发送缓冲区，由 UART 发送完成中断驱动实际输出。不阻塞 CPU uartputc(c) hal_uart.c
hal_getchar() 从 UART 接收寄存器轮询读取一个字符。若无数据则返回 -1 uartgetc() hal_uart.c
hal_console_intr(handler) 注册字符接收的回调函数。当 UART 收到字符触发中断时，调用此 handler 处理输入字符（通常是 consoleintr） uartintr() → 回调 consoleintr() hal_uart.c + console.c
同步 vs 异步输出：内核启动阶段（printfinit 之前）使用同步输出 uartputc_sync，因为此时中断系统未就绪。正常运行时使用中断驱动的异步输出 uartputc，CPU 无需忙等 UART 发送完成，效率更高。
上下文切换 hal/hal_ctx.h

1. 作用

定义上下文切换的数据结构和切换函数。操作系统在多个进程/线程之间切换时，必须保存当前 CPU 的寄存器状态并恢复下一个进程的寄存器状态。hal_context 结构体定义了哪些寄存器需要保存。

2. 关联关系

平台实现：hal/riscv/hal_swtch.S（汇编实现）执行实际的寄存器保存/恢复。结构体 hal_context 中字段的顺序必须与汇编代码中的加载/存储顺序严格一致。

主要调用者：

kernel/proc.c: sched() — 调用 swtch(&p->context, &c->context) 从进程切换到调度器
kernel/proc.c: scheduler() — 调用 swtch(&c->context, &p->context) 从调度器切换到进程

3. 内容列表

声明类型功能实现
struct hal_context 结构体（14 个 uint64 字段） 保存 RISC-V 的 14 个被调用者保存寄存器：ra, sp, s0-s11。这些是 C 函数调用约定要求被调函数保存的寄存器，切换进程时必须保留 hal_ctx.h 定义
hal_switch(**old, *new) 函数声明 保存当前 CPU 寄存器到 *old 指向的 context，从 *new 加载新进程的 context 并跳转。是进程调度的最底层原语 hal_swtch.S 汇编
为什么用汇编实现：C 语言无法直接读写所有寄存器（sp 在函数调用时会改变，ra 会被覆盖）。上下文切换需要在"裸金属"层面精确控制每一个寄存器的保存和恢复时机，因此必须用汇编。hal_ctx.h 提供 C 层面的结构体定义和函数声明，hal_swtch.S 是实际的汇编实现。
跨平台注意事项：LoongArch 的被调用者保存寄存器集合与 RISC-V 不同（寄存器数量、名称、ABI 约定均有差异），需要提供自己的 hal_ctx.h（或条件编译）和对应的 hal_swtch.S。但调度逻辑（proc.c 中的 sched/scheduler）完全不变。

3. "先声明后实现"的设计策略

在 8 个头文件中，可以看到一些 HAL 接口（如 hal_irq_init()、hal_putchar_sync()）声明了但内核代码仍使用旧的函数名（如 plicinit()、uartputc_sync()）直接调用。这不是设计缺陷，而是渐进式移植策略的体现：

  1. 第 2 周：将平台代码从 kernel/ 搬入 hal/riscv/，保持函数名不变，系统可运行。
  2. 第 3 周：定义 HAL 接口头文件——确立每个子系统的"规范函数名"。旧的 RISC-V 命名仍然可用，但新平台（LoongArch）应直接实现 HAL 规范名。
  3. 未来：逐步将内核中的 plicinit() → hal_irq_init() 等调用点替换为 HAL 规范名，完成接口统一。

4. 接口粒度原则

每个 HAL 头文件只定义该子系统最少的必要操作，遵循以下规则：

  • 一个头文件 = 一个硬件子系统。不合并不同职责的接口（如不会把定时器和中断控制器放一起）。
  • 只暴露差异点。页表 walk 算法跨架构相同 → 留在 kernel/vm.c；TLB 刷新指令不同 → 进入 hal_vm.h。
  • 最小接口。hal_intr.h 只有 4 个函数，因为中断控制器的全部职责就这 4 个：初始化（全局 + 每核）、认领、完成。
  • 参数简洁。尽量用基本类型（int, uint64），避免平台相关结构体传入 HAL 接口。

5. 汇编与 C 的分工

文件语言原因
hal_swtch.S汇编上下文切换需要精确控制每个寄存器的保存/恢复时机
hal_tramp.S汇编用户态↔内核态陷入/返回需要操作 CSR 和精确的指令序列
hal_kvec.S汇编内核态中断向量需要保存全部寄存器并跳转到 C 处理函数
