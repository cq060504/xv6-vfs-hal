环境搭建 一、Docker 环境搭建：LoongArch 交叉编译 + QEMU

1.1 为什么用 Docker？

  • 隔离性：LoongArch 工具链与 RISC-V 工具链完全隔离，避免环境冲突
  • 可复现：Dockerfile 即文档，任何人都能重建相同环境
  • WSL2 友好：Docker Desktop 在 WSL2 中运行良好，性能损失很小
  • 一键切换：通过 volume 挂载源码，宿主机编辑，Docker 编译

1.2 Dockerfile — Ubuntu 24.04 + LoongArch 工具链

以下 Dockerfile 基于 Ubuntu 24.04 LTS，通过 apt 安装所有必需工具链：

# ======================== Dockerfile.loongarch ========================
FROM ubuntu:24.04

# 避免交互式安装提示
ENV DEBIAN_FRONTEND=noninteractive

RUN apt-get update && apt-get install -y --no-install-recommends \
    # LoongArch64 交叉编译工具链
    gcc-loongarch64-linux-gnu \
    g++-loongarch64-linux-gnu \
    binutils-loongarch64-linux-gnu \
    # QEMU（含 loongarch64 系统模拟）
    qemu-system-misc \
    qemu-system-data \
    # 调试
    gdb-multiarch \
    # 通用构建依赖（xv6 需要）
    make \
    python3 \
    perl \
    git \
    build-essential \
    # 工具
    vim \
    xxd \
    && rm -rf /var/lib/apt/lists/*

# 工作目录

# 默认命令
CMD ["/bin/bash"]

步骤 3（推荐）：一键构建并运行脚本

创建 scripts/docker-loongarch.sh 脚本，方便反复使用：

#!/bin/bash
# scripts/docker-loongarch.sh — 启动 LoongArch 开发容器
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
PROJECT_DIR="$(dirname "$SCRIPT_DIR")"

IMAGE_NAME="xv6-loongarch"
CONTAINER_NAME="xv6-la-dev"

# 首次运行：构建镜像
if ! docker image inspect "$IMAGE_NAME" &>/dev/null; then
    echo "Building Docker image $IMAGE_NAME..."
    docker build -t "$IMAGE_NAME" -f "$PROJECT_DIR/Dockerfile.loongarch" "$PROJECT_DIR"
fi

# 如果容器已存在则 attach，否则新建
if docker container inspect "$CONTAINER_NAME" &>/dev/null 2>&1; then
    docker start -ai "$CONTAINER_NAME"
else
    docker run -it --name "$CONTAINER_NAME" \
        -v "$PROJECT_DIR:/xv6" \
        "$IMAGE_NAME" /bin/bash
fi

2.2 特权级架构详解

PLV0（内核态）vs PLV3（用户态）

LoongArch 只有两级特权级。系统启动后即处于 PLV0（最高特权级），用户态程序在 PLV3。

这意味着没有了 RISC-V 的 M 态，所有初始化直接在 PLV0 完成，简化了 start.c 的启动流程。

操作RISC-V 做法LoongArch 做法
进入内核M 态 start() → mret → S 态 main()PLV0 直接启动，无需切换
用户→内核ecall in U-mode → stvec → trampolinesyscall in PLV3 → EENTRY → trampoline
内核→用户sret (S→U)ertn (PLV0→PLV3)
PMPpmpcfg0/pmpaddr0不需要，DMW 替代
中断委托medeleg/mideleg（M→S）不需要（无 M 态，中断直达 PLV0）
简化点：LoongArch 的 hal_start.c 比 RISC-V 简单很多——不需要 M 态配置（无 mstatus/mepc/medeleg/mideleg/pmpcfg），直接配置 CRMD、EENTRY、PGDL 等即可。

2.6 软件 TLB 重填机制（最重要差异）

核心差异：RISC-V 的 MMU 在 TLB miss 时自动执行硬件页表 walk（3 级），而 LoongArch 的 MMU 在 TLB miss 时触发 TLB 重填异常，由内核的 TLBRENTRY 处理函数通过软件完成页表 walk 和 TLB 填入。

这意味着：

  1. kernel/vm.c 中的 walk() 函数不再被 MMU 硬件调用，而是被 tlb_refill_handler 调用
  2. 需要在 hal_tramp.S 中实现 TLBRENTRY 入口（独立于普通异常入口 EENTRY）
  3. TLB 填入指令：tlbwr（随机写入）、tlbfill（填入指定项）、tlbsrch（搜索）
  4. 需要 TLBIDX CSR 管理 TLB 索引
// LoongArch TLB 重填伪代码
tlb_refill_handler:
    // 1. 从 BADV 读取引发 TLB miss 的虚拟地址
    csrrd  t0, 0x7          // BADV → t0

    // 2. 软件 walk 页表（类似 RISC-V walk()）
    csrrd  t1, 0x19         // PGDL → t1 (页表基址)
    //   提取 VA[38:30] → PGD index → 加载 PGD entry
    //   提取 VA[29:21] → PMD index → 加载 PMD entry
    //   提取 VA[20:12] → PTE index → 加载 PTE

    // 3. 填充 TLB（使用 tlbfill 或 tlbwr）
    csrwr  t2, 0x88        // TLBELO0 (页表项低64位)
    csrwr  t3, 0x89        // TLBELO1 (页表项高64位)
    csrwr  t4, 0x8c        // TLBEHI (虚拟地址高 + ASID)
    tlbwr                    // 写入随机 TLB 项

    // 4. 返回（ertn）
    ertn
简化策略（推荐用于第5周）：利用 LoongArch 的 DMW（Direct Mapping Window） 功能。
DMW 提供无需 TLB 的直接虚拟→物理地址映射，通过 4 个 DMW 配置寄存器（DMW0-DMW3）配置。
例如：配置 DMW0 将内核虚拟地址 0x9000000000000000-0x9000FFFFFFFFFFFF 直接映射到物理地址 0x00000000-0xFFFFFFFFFFFF。
这样内核代码和数据访问不需要 TLB，只需为用户态地址实现 TLB 重填。这与 RISC-V 的 Sv39 效果一致。

arch.h 四、hal/loongarch/arch.h — CSR 操作 & PTE 格式

这是 16 个文件中最重要的一个。它提供所有 CSR 内联函数和 PTE 宏，被内核所有文件通过 hal/hal.h 间接包含。

CSR 内联函数模板（LoongArch CSR 指令格式）

// LoongArch CSR 访问指令格式：
//   csrrd rd, csr_num     — 读 CSR → GPR
//   csrwr rd, csr_num     — 写 GPR → CSR
//   csrxchg rd, rj, csr_num — 交换: rd ← CSR, CSR[被 rj 选择的位] ← rd[对应位]

// ============ 读 CSR 模板 ============
static inline uint64 r_crmd(void) {
    uint64 x;
    asm volatile("csrrd %0, 0x0" : "=r"(x));  // CRMD = 0x0
    return x;
}

// ============ 写 CSR 模板 ============
static inline void w_crmd(uint64 x) {
    asm volatile("csrwr %0, 0x0" : : "r"(x));
}

// ============ CSR 位交换模板 ============
// csrxchg: 将 CSR 中被 rj 的bit=1 对应的位替换为 rd 的对应位
// 常用于原子开启/关闭中断
static inline uint64 csr_xchg_crmd(uint64 mask, uint64 val) {
    uint64 old;
    // csrxchg rd, rj, csr:   old = CSR; CSR = (CSR & ~mask) | (val & mask)
    asm volatile("csrxchg %0, %2, 0x0" : "=r"(old), "+m"(*(char(*)[8])"") : "r"(mask | (val << 32 ? 0 : 0)));
    return old;
}
注意：csrxchg 的语义比较微妙。推荐使用更简单的方式实现中断控制。
csrxchg rd, rj, csr：rd = old_CSR; CSR = (CSR & ~rj) | (rd & rj).
更安全的做法是读-改-写：csrrd → 修改 → csrwr。

#define PGSHIFT 12

// LoongArch PTE flags
#define PTE_V    (1L << 0)   // Valid
#define PTE_D    (1L << 1)   // Dirty
// PTE_PLV: bits 2-3, 0=PLV0, 3=PLV3 (user accessible = PLV3)
#define PTE_PLV0 (0L << 2)
#define PTE_PLV3 (3L << 2)   // user-mode accessible
#define PTE_MAT_CC (1L << 4) // MAT=1 (cache coherent)
#define PTE_G    (1L << 6)   // Global
#define PTE_NX   (1L << 60)  // No Execute
#define PTE_RPLV (1L << 61)  // bit=1 -> not readable by lower PLV → prev write-only

// 权限组合 (match RISC-V semantics)
#define PTE_R (PTE_RPLV)    // When RPLV=1, lower PLV cannot read → acts as "write-only" protect
#define PTE_W (PTE_D)        // D bit enables writes
#define PTE_X (0)            // executable unless NX is set
#define PTE_U (PTE_PLV3)     // accessible from PLV3 (user)

// PA ↔ PTE 转换（LoongArch PPN 分两段）
// PTE[59:12] = PPN[47:0], PTE[11:7] = PPN[52:48]
#define PA2PTE(pa)  ((((uint64)(pa) >> 12) << 12) | (((uint64)(pa) >> 48) << 7))
#define PTE2PA(pte) ((((pte) >> 12) << 12) | (((pte) >> 7) << 48))

#define PXMASK          0x1FF
#define PXSHIFT(level)  (PGSHIFT + (9 * (level)))
#define PX(level, va)   ((((uint64)(va)) >> PXSHIFT(level)) & PXMASK)

// MAXVA: LA64 supports up to 2^48 (4-level), but we use 3-level for xv6
#define MAXVA (1L << (9 + 9 + 9 + 12 - 1))

#define MAKE_SATP(pagetable) ((uint64)(pagetable))  // no mode bits in PGDL

entry.S 六、hal/loongarch/hal_entry.S — 启动入口

与 RISC-V 的关键区别：

  • LoongArch 启动时直接处于 PLV0（最高特权），无需 M 态切换到 S 态
  • 加载地址取决于链接脚本（kernel.ld 的 . 起始地址）
  • 通过 CPUID CSR 获取核心 ID，而非 mhartid
  • 指令助记符不同：la（LoongArch）vs la（RISC-V）通用，jirl 跳转，b 无条件分支
# hal/loongarch/hal_entry.S
# 启动入口 — 设置栈，跳转到 start()

.section .text.entry
.global _entry
_entry:
    # 设置栈: sp = stack0 + ((cpuid + 1) * 4096)
    # stack0 在 hal_start.c 中声明
    la.abs  $sp, stack0          # 加载 stack0 的绝对地址
    li.d    $t0, 4096
    csrrd   $t1, 0x20            # CPUID CSR = 0x20
    addi.d  $t1, $t1, 1
    mul.d   $t0, $t0, $t1
    add.d   $sp, $sp, $t0

    # 跳转到 C 的 start()
    bl      start                 # branch and link (call)

spin:
    b       spin                  # infinite loop (should not reach)
注意：la.abs 使用绝对地址（区别于 la.pcrel 使用 PC 相对地址）。
如果链接脚本将 .text.entry 放在 0x1c000000，则 la.abs 可以正确加载符号地址。
start.c 七、hal/loongarch/hal_start.c — 平台初始化

RISC-V 的 start() 需要配置 M 态（mstatus, mepc, medeleg, mideleg, pmp），然后 mret 切换到 S 态。
LoongArch 启动时已在 PLV0，无需切换特权级，初始化流程更简洁。

LoongArch start() 需要做的事：

  1. 配置 CRMD：关闭 DA（直接地址翻译）、关闭 PG（页表映射）——先以物理地址运行
  2. 设置 EENTRY = kernelvec（内核态中断入口）
  3. 使能定时器中断（ECFG.LIE[11]=1）
  4. 初始化定时器（TCFG + TVAL）
  5. 将 cpuid 保存到 tp 寄存器
  6. 跳转到 main()
// hal/loongarch/hal_start.c
#include "types.h"
#include "param.h"
#include "hal/hal.h"
#include "defs.h"

void main();
void timerinit();

__attribute__((aligned(16))) char stack0[4096 * NCPU];

void start()
{
    // 1. 加电默认 PLV0，无需切换

    // 2. 确保 MMU 关闭（直接地址模式）
    uint64 crmd = r_crmd();
    crmd &= ~(1L << 4);   // CRMD.PG = 0 (disable paging)
    crmd &= ~(1L << 3);   // CRMD.DA = 0 (disable direct addr translation)
    w_crmd(crmd);

    // 3. 设置异常入口
    w_eentry((uint64)kernelvec);

    // 4. 初始化定时器
    timerinit();

    // 5. 保存 cpuid 到 tp
    int id = r_cpuid();
    asm volatile("or $r2, %0, $r0" : : "r"(id));  // tp = cpuid

    // 6. 直接跳转到 main()（无需 mret）
    main();
}

// 定时器初始化
void timerinit()
{
    // 使能定时器中断 (ECFG.LIE bit 11 = 1)
    w_ecfg(r_ecfg() | (1L << 11));

    // 配置 TCFG: 定时器使能 + 周期模式
    // TCFG[0] = 1 (enable), TCFG[4] = 0 (periodic auto-reload off)
    w_tcfg(0x1);

    // 设置第一次定时器中断（递减值约 1/10 秒）
    w_tval(1000000);
}

// HAL 包装
void hal_timer_init(void) { timerinit(); }
与 RISC-V 的差异：LoongArch 的 start() 没有 mepc/mret/mstatus/pmpcfg/medeleg/mideleg 等冗长的 M 态配置。
LoongArch 也有 M 态（PLV0），但硬件加电后直接处于 PLV0，不需要"提升"到更高特权级再做初始化。
tramp.S 八、hal/loongarch/hal_tramp.S — 用户态/内核陷入

这是移植过程中最复杂的文件。需要实现三个入口：

  1. uservec：用户态陷入时的入口（类似 RISC-V trampoline 的 uservec）
  2. userret：内核返回用户态的出口（类似 RISC-V trampoline 的 userret）
  3. tlb_refill：TLB 重填异常入口（在 LoongArch 上是独立入口 TLBRENTRY，不使用 EENTRY）

LoongArch 异常入口机制

入口 CSR用途RISC-V 对应
EENTRY (0xC)普通异常/中断入口stvec
TLBRENTRY (0x88)TLB 重填异常入口(无—RISCV 硬件 walk)
MERRENTRY (0x93)机器错误异常入口(无—相当于 NMI)

uservec 实现要点

# hal/loongarch/hal_tramp.S
# 用户态陷入/返回 trampoline

#include "arch.h"
#include "memlayout.h"

.section trampsec
.globl trampoline
.globl usertrap

trampoline:
.align 4
.globl uservec
uservec:
    # 当用户态（PLV3）发生异常时，硬件:
    #   1. PRMD.PPLV = CRMD.PLV (=3, 用户态)
    #   2. PRMD.PIE = CRMD.IE
    #   3. CRMD.PLV = 0 (强制进入 PLV0)
    #   4. CRMD.IE = 0 (关中断)
    #   5. ERA = 触发异常的指令 PC
    #   6. PC = EENTRY (即 uservec)

    # 此时栈是用户态栈！不能使用栈操作保存寄存器
    # 必须通过 TRAPFRAME 保存寄存器状态
    # 使用 KScratch 保存临时值

    # 交换 a0 和 KScratch（保存 a0，取得 TRAPFRAME 地址）
    csrwr  $a0, 0x99          # KScratch0 → a0 (KScratch0 存了 TRAPFRAME)

    # 现在 a0 = TRAPFRAME
    # 保存所有通用寄存器到 TRAPFRAME

    # ... 与 RISC-V trampoline.S 类似的寄存器保存逻辑 ...
    # 但使用 LoongArch 指令: st.d (store doubleword)

    # st.d ra, a0, offset
    st.d    $ra, $a0, 40
    st.d    $sp, $a0, 48
    # ... 所有 32 个 GPR ...

    # 从 TRAPFRAME 恢复 a0
    csrwr  $t0, 0x99
    st.d    $t0, $a0, 112      # a0 → TRAPFRAME.a0

    # 加载内核栈
    ld.d    $sp, $a0, 8        # sp = p->trapframe->kernel_sp
    ld.d    $r2, $a0, 32       # tp = p->trapframe->kernel_hartid

    # 加载 usertrap() 地址
    ld.d    $t0, $a0, 16       # p->trapframe->kernel_trap

    # 加载内核页表
    ld.d    $t1, $a0, 0        # p->trapframe->kernel_satp

    # 刷新 TLB + 切换页表
    invtlb  0, $r0, $r0
    csrwr  $t1, 0x19          # w_pgdl(t1)
    invtlb  0, $r0, $r0

    # 跳转到 usertrap()
    jirl   $r0, $t0, 0

.globl userret
userret:
    # 从 usertrap() 返回，a0 = 用户页表基址

    # 切换页表
    invtlb  0, $r0, $r0
    csrwr  $a0, 0x19          # w_pgdl(a0)
    invtlb  0, $r0, $r0

    # 从 TRAPFRAME 恢复所有寄存器
    li.d    $a0, TRAPFRAME
    # ... ld.d 恢复所有寄存器 ...

    # 设置 ERA（异常返回地址 = user pc）
    # 设置 PRMD.PPLV = 3 (用户态), PRMD.PIE = 1 (开中断)
    # 执行 ertn 返回到用户态
    csrwr  $t0, 0x6           # w_era(t0)
    csrwr  $t1, 0x1           # w_prmd(t1)
    ertn
最困难的部分：LoongArch 使用软件 TLB 重填，这意味着还需要实现 tlb_refill 处理函数。
当 TLB miss 时，CPU 跳转到 TLBRENTRY CSR 指向的地址（而非 EENTRY）。
此处理函数需要在软件中执行页表 walk，然后用 tlbwr/tlbfill 填充 TLB。
推荐：第 5 周先用 DMW 映射内核空间，避免内核 TLB miss；第 6 周再实现完整的 TLB refill。

直接复用：UART 驱动的逻辑是平台无关的（都是 16550a 标准接口），只需通过 memlayout.h 提供正确的 UART0 基地址。
LoongArch QEMU virt 的 UART0 地址是 0x1FE001E0，而 RISC-V virt 是 0x10000000。
virtio.c 十三、hal/loongarch/hal_virtio.c — 磁盘驱动
关键差异：RISC-V QEMU virt 使用 MMIO virtio 设备（基地址 0x10001000），而 LoongArch QEMU virt 使用 PCI virtio 设备。
这意味着不能直接复用 RISC-V 的 MMIO virtio 驱动，需要实现 PCI virtio 接口。
替代方案（推荐）：第 5-6 周用 ramdisk 代替 virtio 磁盘，先用 -initrd 挂载文件系统；第 7 周再实现 PCI virtio。

如果决定直接支持 PCI virtio，需要：

  1. PCI 总线枚举（至少找到 virtio-blk 设备的 BAR）
  2. MSI-X 或 INTx 中断配置
  3. virtio 协议（与 MMIO 版基本一致，仅传输层不同）
// PCI virtio 简化路径 (参考 Linux arch/loongarch 的 PCI 初始化)
// 1. QEMU LoongArch virt 的 PCIe 控制器在 MMIO 区域
//    基地址: 0x40000000 (GPEX host bridge)
// 2. 扫描 PCI 总线，找到 VendorID=0x1af4 (Virtio) 的设备
// 3. 配置 BAR 地址、中断线
// 4. 后续 virtio 协议与 MMIO 版本相同
kernel.ld 十四、hal/loongarch/kernel.ld — 链接脚本 
/* hal/loongarch/kernel.ld */
OUTPUT_ARCH("loongarch")
ENTRY(_entry)

SECTIONS
{
  /*
   * QEMU LoongArch virt machine boots at 0x1c000000
   * (reset vector in firmware/BIOS region).
   * With -bios none or direct loading, we put kernel here.
   * Adjust as needed based on boot strategy.
   */
  . = 0x1c000000;

  .text : {
    *(.text.entry)
    *(.text .text.*)
    . = ALIGN(0x1000);
    _trampoline = .;
    *(trampsec)
    . = ALIGN(0x1000);
    ASSERT(. - _trampoline == 0x1000, "error: trampoline larger than one page");
    PROVIDE(etext = .);
  }

  .rodata : {
    . = ALIGN(16);
    *(.rodata .rodata.*)
  }

  .data : {
    . = ALIGN(16);
    *(.data .data.*)
  }

  .bss : {
    . = ALIGN(16);
    *(.bss .bss.*)
  }

  PROVIDE(end = .);
}

附录 关键：kernel/trap.c 中的 scause 判断适配

kernel/trap.c 中的 devintr() 函数直接比较 scause 的常数值来判断中断类型。由于 LoongArch 使用 ESTAT 而非 scause，需要处理这个差异。

方案 A：在 arch.h 中定义等价宏（推荐）

// hal/loongarch/arch.h — 在 hal_read_scause() 中转换格式
// 将 ESTAT 转换为类似 RISC-V scause 的编码，让 trap.c 无需修改

// LoongArch ESTAT 到 RISC-V scause 的映射
#define LA_SYS      (11)         // EXCCODE_SYS = 11
#define LA_TLBL     (1)          // Load TLB miss
#define LA_TLBS     (2)          // Store TLB miss

// 将 ESTAT 转换为 scause-like 值
static inline uint64 estat_to_scause(uint64 estat) {
    uint64 ecode = (estat >> 16) & 0x3f;    // ESTAT.Ecode
    uint64 is_bits = estat & 0x1fff;          // ESTAT.IS[12:0]

    // 定时器中断: IS[11] = 1
    if (ecode == 0 && (is_bits & (1 << 11))) {
        return 0x8000000000000005ULL;  // RISC-V timer interrupt
    }
    // 外部 HWI 中断: IS[12] = 1
    if (ecode == 0 && (is_bits & (1 << 12))) {
        return 0x8000000000000009ULL;  // RISC-V external interrupt
    }

    // 异常: 映射 Ecode 到 scause
    switch (ecode) {
        case 11: return 8;   // SYS → syscall
        case 1:  return 13;  // TLBL → load page fault
        case 2:  return 15;  // TLBS → store page fault
        case 3:  return 12;  // TLBI → instruction page fault
        default: return ecode;
    }
}

// 重写 hal_read_scause()
static inline uint64 hal_read_scause(void) {
    return estat_to_scause(r_estat());
}
这样 trap.c 中的比较代码不需修改：hal_read_scause() == 8、scause == 0x8000000000000005L 等比较仍然有效。
这是 HAL 设计的价值所在——平台差异封装在接口实现中，调用方无感知。
