# 技术难点分析：TLB Refill 和 MAXVA — 为什么之前没能解决

## 一、对比：之前版本 vs 最终修复版本

### 1.1 之前版本（commit `765c76f`）的 TLB handler

**之前版本使用的是纯软件 `ld.d` walk（3级从PGDL直接取L2）**：

```asm
# 3-level software walk: PGDL → L2(VA[38:30]) → L1(VA[29:21]) → L0(VA[20:12])
csrrd $t0, 0x7         # BADV — 错误！不是TLBRBADV
csrrd $t1, 0x19        # PGDL
srli.d $t2, $t0, 30    # L2 index
... ld.d $t2            # 读PTE
srli.d $t1, $t2, 12    # 提取PPN → L1 base
slli.d $t1, $t1, 12
# 重复 L1 → L0
# 直接写 leaf PTE 到 TLBRELO0 (未清除P/W位!)
csrwr $t2, 0x12        # TLBRELO0 = 原始PTE
```

### 1.2 最终修复版本（commit `c9e1c32`）的 TLB handler

**最终版本使用正确的 `lddir`/`ldpte` 指令（4级从PGDL走L3->L2->L1->L0）**：

```asm
# 4-level lddir walk: PGDL → L3(VA[47:39]) → L2(VA[38:30]) → L1(VA[29:21]) → L0
csrrd $t0, 0x89        # TLBRBADV — 正确
csrrd $t0, 0x19        # PGDL

# 边界检查：拒绝VA>=MAXVA和VA<2*PGSIZE
srli.d $t0, $t0, 38
bnez $t0, .Ltlbr_invalid

# L3 walk
lddir $t0, $t0, 3      # 硬件walk（使用PWCL/PWCH配置）
andi $t0, $t0, 1       # 检查V
beqz $t0, .Ltlbr_invalid
csrrd $t0, 0x19        # 逐级重读PGDL再走（确保base正确）
lddir $t0, $t0, 3
addi.d $t0, $t0, -1    # 清除V标志

# L2 walk (类似)
# L1 walk (类似)

ldpte $t0, 0           # leaf PTE → TLBRELO0
ldpte $t0, 1           # leaf PTE → TLBRELO1
# 清除P/W位（bit7,bit8是TLBELO保留位）
bstrins.d $t0, $r0, 8, 7
csrwr $t0, 0x8c
tlbfill
```

## 二、之前版本的根本性错误（逐条分析）

### 错误 1：3级 walk 与 4 级页表不匹配 ⭐⭐⭐⭐⭐

**这是最致命的错误。**

之前 `765c76f` 版本中：
- `walk()` 函数（`kernel/vm.c`）创建的是 **4 级**页表结构：PGD→PUD→PMD→PTE
- KVAS 中 PWCL/PWCH 使用的也是 4 级配置（默认或显式）
- 但 TLB handler 走的是 **3 级** walk：直接从 PGDL 取 L2（跳过 L3）

**后果**：TLB handler 从 PGD 页取 PTE 时，把 L3（PGD）的 entry 当成了 L2（PUD）entry 来处理。整个 walk 偏移了一级——读到的是错误的 PPN、走了错误的下一级基址。

**为什么之前没发现**：因为 `ld.d` 软 walk 在 3 级和 4 级之间切换时不会报错——它能走到最后取到 leaf PTE，只是 PPN 不对。症状是"用户态写入不生效"而非"系统崩溃"，这是一个**静默数据损坏**问题，极难定位。

**为什么之前没修好**：我们在对话中先后尝试了：
1. `ld.d` 3级 walk → 数据不一致（PPN 偏移）
2. `ld.d` 4级 walk → DA=1下用栈保存寄存器 → 栈在高VA无RAM → hang
3. CSR 保存寄存器的 4级 walk → `bstrins.d` 兼容性问题
4. 各种 dbar/invtlb/PTE mask 修复 → 都是针对症状而非根因

**始终没意识到要回到 `lddir`/`ldpte`**。最终修复用的是 `lddir`/`ldpte` 指令（手册规定的标准做法），配合 PWCL/PWCH 配置，让硬件自动按 4 级 walk。

### 错误 2：没有清除 P/W 位 ⭐⭐⭐⭐

内存 PTE 中 bit7=P（硬件存在标记）、bit8=W（硬件可写标记）。在 TLBELO（TLB entry low）寄存器中，这两个 bit 是**保留位**，应当为 0。

之前版本直接把内存 PTE 原样写入 TLBELO0：
```asm
csrwr $t2, 0x12      # TLBRELO0 = raw PTE（含P=1, W=1）
```

导致 TLB 条目被污染：保留位被设为 1，TLB 查找产生未定义行为。

最终修复版本在 `tlbfill` 前加：
```asm
csrrd   $t0, 0x8c
bstrins.d $t0, $r0, 8, 7    # 清零 bits[8:7]
csrwr   $t0, 0x8c
```

### 错误 3：MAXVA 从 1<<38 改为 1<<46 ⭐⭐⭐⭐

在 `765c76f` → `80dd1c3` 之间，有人将 MAXVA 从 `1<<38` 改为 `1<<46`。这是对 LoongArch 48-bit VA 能力的"过度利用"——以为更大的 VA 空间能解决地址映射问题。

**背景**：`765c76f` 版本 kinit 报 `panic: mappages: va not aligned`。根因是 trampoline/trapframe/KSTACK 的 VA 与空闲 RAM identity-map 区域冲突。当时的"修复"是扩大 MAXVA。

**实际后果**：
- `usertests` 中 `MAXVAplus` 测试期望 `VA >= 1<<38` 的访问失败
- MAXVA=1<<46 让这些测试地址变成了"合法"地址
- LoongArch 硬件对超出 VALEN(48) 的高 VA 做截断 → 高 VA 别名到低地址 → 用户非法访问了低地址的数据
- `MAXVAplus` 测试失败：`oops wrote 0x0001000000000000`

**正确的修复**：
1. 恢复 MAXVA=1<<38（与 xv6 原始 Sv39 一致）
2. **在 TLB handler 入口处做边界拒绝**（在 lddir 之前！）
3. **exec 侧映射低2页为非用户 guard**（防止高地址别名命中）
4. **RVACFG=8** 硬件辅助降低有效 VA 位宽
5. LoongArch 用户程序从 `0x2000` 链接

### 错误 4：读取 BADV 而非 TLBRBADV ⭐⭐

之前版本读 `csrrd $t0, 0x7`（BADV = Bad Virtual Address），但 TLB refill 的故障地址在 `TLBRBADV`（CSR 0x89）。

虽然在某些情况下 BADV 和 TLBRBADV 值相同，但严格来说，TLB refill handler 应该读 TLBRBADV。

### 错误 5：vmfault 填 TLB 被 userret 冲掉 ⭐⭐⭐

我们花大量时间尝试在 `vmfault` 中直接手工填 TLB，但始终被 `userret` 中的 `invtlb 0, $r0, $r0` 冲掉。

正确的方法应该是让 TLB refill handler 本身正确工作，而不是事后修补。

### 错误 6：PWCL/PWCH 未显式配置 ⭐⭐⭐

`765c76f` 版本的 `kvminithart()` 没有显式设置 PWCL/PWCH，依赖 QEMU 默认值。

加上我们之前尝试手动设置 PWCL/PWCH 为 4 级时，因赋值和位掩码错误，导致配置与实际页表结构不符。

最终版本在 `kvminithart()` 中正确配置：
```c
asm volatile("csrwr %0, 0x1C" : : "r"(
  (12UL<<0)|(9UL<<5)|(21UL<<10)|(9UL<<15)|(30UL<<20)|(9UL<<25)  // PWCL
));
asm volatile("csrwr %0, 0x1D" : : "r"((39UL<<0)|(9UL<<6)));  // PWCH
```

## 三、经验教训

| 维度 | 之前的误区 | 正确的做法 |
|---|---|---|
| **TLB walk 方法** | 自己写 ld.d 软walk，3级/4级切换出错 | 用 lddir/ldpte 标准指令，在 kvminithart 配 PWCL/PWCH |
| **MAXVA 策略** | 放宽 MAXVA 以适配 LoongArch 宽 VA | 严格保持 xv6 原始 MAXVA=1<<38，多层防护 |
| **TLB entry 格式** | 内存 PTE 直接写 TLBELO | 清除 P/W 保留位 |
| **TLB handler 输入** | 读 BADV | 读 TLBRBADV |
| **调试方法** | vmfault 手工填 TLB → userret 冲掉 | 直接 fix TLB handler |
| **问题定位** | 在症状层面加各种 patch（dbar/invtlb） | 追根因：页表结构与 handler 不一致 |

## 四、最终修复的核心设计理念

1. **TLB handler 使用 `lddir`/`ldpte`**：这是 LoongArch 手册规定的标准做法，无需自己实现软 walk
2. **PWCL/PWCH 必须在 `kvminithart()` 中显式配置**：保证硬件 walker 和内核 walk() 使用相同的页表索引
3. **MAXVA 保持 1<<38**：不宽于 xv6 原始 Sv39 契约
4. **三层 MAXVA 防护**：
   - C 路径：`vmfault()` 拒绝 `va >= MAXVA`
   - TLB refill：`lddir` 前检查边界
   - 硬件辅助：`RVACFG=8` 降低有效 VA
5. **低页 guard**：exec 映射低 2 页为非用户页，防止高地址别名
