# xv6-riscv LoongArch HAL 移植 — 技术难点与解决方案

## 一、项目概述

目标：为 xv6-riscv 第5版增加硬件抽象层（HAL），使其同时支持 RISC-V 和 LoongArch 两个架构，通过 `python3 test-xv6.py -q usertests`。

**当前状态**：RISC-V HAL 已全部完成（`usertests` 通过）。LoongArch HAL 已完成并通过 `usertests -q`。

## 二、总体技术挑战

LoongArch 与 RISC-V 有 6 个架构级关键差异需要 HAL 化：

| 维度 | RISC-V (rv64) | LoongArch (LA64) |
|---|---|---|
| 特权级 | M/S/U 三级 | PLV0(内核)/PLV3(用户) 两级 |
| 页表 walk | 硬件自动（Sv39） | **软件 TLB 重填** |
| TLB 刷新 | `sfence.vma` | `invtlb 0, $r0, $r0` |
| 异常返回 | `sret`/`mret` | `ertn` |
| 中断控制器 | PLIC (MMIO) | EIOINTC (MMIO) |
| 定时器 | CLINT MMIO | TCFG/TVAL CSR（纯 CSR） |

## 三、开发阶段与关键问题

### 第4-5周：环境搭建与基础 HAL

#### 问题 1：工具链与 QEMU 环境兼容性

- Docker 中交叉编译器需指定版本（`gcc-14-loongarch64-linux-gnu`）
- QEMU `-bios` 只接受 raw binary，需 `objcopy -O binary`
- QEMU LoongArch virt 要求至少 1GB RAM
- `-mcmodel=normal`（RISC-V 用 `medany`）

#### 问题 2：内核 `.data/.bss` 在只读 flash 中 ⭐⭐⭐⭐⭐

**这是整个移植过程中最关键的 bug。**

**现象**：`kerneltrap: not from supervisor mode` panic。

**根因分析过程**：
1. 在 main() 各 init 函数前后插入直接 UART 写标记，逐步定位到 `consoleinit()` 后无输出
2. 用 `for(;;)` 暂停测试确认 consoleinit() 本身不 crash
3. 用 GDB 远程调试捕获到确切 panic 位置：`trap.c` 的 `if((sstatus & SSTATUS_SPP) == 0) panic()`
4. 追踪到 `initlock(&cons.lock)` 写 `.bss` 段时触发存储异常

**根本原因**：QEMU `-bios` 将整个内核二进制加载到 flash 区域（0x1C000000，**只读**），但链接脚本将 `.bss` 和 `.data` 放在同一区域内。内核写全局变量时触发存储异常。

**解决方案**：分离 flash/RAM 布局
```
0x1C000000 Flash（只读）: .text, .rodata, .eh_frame
────────────────────────── _data_lma: .data 初始值存于 flash
0x00400000 RAM（可读写）: .data, .bss
```
`start()` 中将 `.data` 初始值从 flash 复制到 RAM，清零 `.bss`。

### 第6周：中断、定时器、上下文切换

#### 问题 3：定时器中断风暴

**现象**：`timerinit()` 后 CPU 陷入无限中断循环。

**根因**：`kernelvec` 最初只有 `ertn` 一条指令。`ertn` 返回后定时器中断标志未清除（未写 `TICLR`），CPU 立即再次响应同一中断。

**解决**：在 `clockintr()` 中调用 `w_ticlr(0)` 清除中断标志。

#### 问题 4：UART 初始化与 QEMU 兼容性

- 波特率重编程破坏输出：QEMU virt UART 使用默认波特率，改变导致乱码
- FCR FIFO 重置触发 QEMU 内部断言崩溃（QEMU 8.2 的 bug）
- TX idle 轮询不可靠：QEMU LSR 寄存器行为与标准 16550a 不同

**解决**：`uartinit()` 跳过 FCR 和波特率操作，只做 `initlock`。

#### 问题 5：SSTATUS_SPP/SPIE → PRMD 转换

**根因**：RISC-V 使用 `sstatus.SPP/SPIE` 判断 trap 来源，LoongArch 使用 `PRMD.PPLV/PIE`。两者位序完全不同。

**解决**：新增 `r_sstatus_compat()` 和 `w_sstatus_compat()` 转换函数，将 LoongArch PRMD 格式映射为 RISC-V sstatus 格式，让 `trap.c` 通用代码无需修改。

#### 问题 6：PHYSTOP 范围与 kmap 冲突

**现象**：`panic: mappages: remap`

**根因**：`etext`（0x1C009000，flash）和 `PHYSTOP`（0x00400000+RAM）之间的 `kvmmap` 范围计算为负数→无符号下溢→映射越界。

#### 问题 7：MAXVA 宏类型溢出

`#define MAXVA (1L << 38)` 中 `1L` 触发 GCC `-Werror=shift-count-overflow`。

**解决**：改用 `1ULL`。

### 第7周：TLB 重填 — 最复杂的子系统

#### 问题 8：TLB refill handler 的设计 ⭐⭐⭐⭐⭐

**背景**：LoongArch 没有硬件页表 walk。TLB miss 时，CPU 触发 TLB 重填异常，跳转到 `TLBRENTRY`（CSR 0x88），由软件完成页表 walk 并填入 TLB。

**Handler 执行环境**（关键约束）：
- CPU 自动设置 `DA=1, PG=0`（直接地址翻译模式，VA=PA）
- 不能使用内核栈（`$sp` 指向用户栈，DA=1下 VA=PA，用户栈的物理地址可能无 RAM）
- 只能用 CSR 保存临时寄存器（`TLBRSAVE`）
- 只能访问物理地址空间的页表页
- 使用 `lddir/ldpte` 指令走页表，`tlbfill` 填入 TLB

**Handler 最终实现**（`hal/loongarch/hal_tlbrefill.S`）：

1. 保存 `$t0` 到 `TLBRSAVE`
2. 设置 `TLBREHI.PS = 12`（4KB 页）
3. **边界检查**：拒绝 `VA >= MAXVA(1<<38)` 和 `VA < 2*PGSIZE`
4. 用 `lddir` 走 4 级页表（从 PGDL 读 PGD，逐一走 PUD→PMD→PTE）
5. 每级检查 V 位，任何一级无效则走 `Ltlbr_invalid` 路径
6. 用 `ldpte` 读取 leaf PTE 到 TLBRELO0/1
7. **清除 P/W 位**（内存 PTE 的 bit7/8 在 TLBELO 中是保留位）
8. `tlbfill` 填入随机 TLB 槽
9. `ertn` 返回

**关键设计决策**：
- 在 `lddir` 前做边界拒绝，防止被 LoongArch 页表 walker 截断到低位后命中错误的页表项
- P/W 位必须清除：内存 PTE 格式中 bit7=P（硬件存在）, bit8=W（硬件可写），这些在 TLBELO 中是保留位，不清除会污染 TLB 条目
- ASID 使用 0（单地址空间），所有进程共享

#### 问题 9：TLB 与页表不一致 — 用户态写入无效 ⭐⭐⭐⭐⭐

**现象**：`copyinstr2` 测试失败。子进程通过 TLB 写入 `static char big[]` 后，内核通过 `walkaddr()`（软件页表 walk）读到仍是 0。用户态写入"消失了"。

**调试过程**：
1. 内核通过 DMW0 直接在 fork 的子进程物理页上写 0xAB/0xCD 并读回 → 成功。确认内核侧页表正确且物理页可访问。
2. 用户态 fill loop 写完 'x' 后立即 `dbar 0` 读回 → 仍读到 0。说明用户态 store 指令根本没写到目标物理页。
3. 排除 DMW0 PLV3 配置（已是 0，用户必须通过 TLB）。
4. 排除非叶 PTE 缺少 P 位（lddir 只检查 V 位，不依赖 P 位）。
5. 发现 walk() 创建的是 4 级页表，TLB handler 最初走 3 级 → **页表级数不匹配**！

**TLB handler 与 walk() 不匹配 — 最终解决**：

- 在 `kvminithart()` 中通过 `PWCL/PWCH` CSRs 显式配置 4 级页表（9-9-9-9-12）
- TLB handler 改为完整 4 级 walk（L3→L2→L1→L0），与 `walk()` 完全一致
- 每级在 `lddir` 后先检查 V 位，再 `addi.d -1` 清除 V 标志得到纯物理地址
- 注意：`lddir` 结果的 V 标志在 bit 0，`addi.d -1` 将其清零。但 `addi.d` 只影响 bit 0，不影响 PPN。对于物理地址 $2^{12}$ 对齐的情况，bit 0 始终为 0，减 1 等于减 1 物理地址出错了！

**`addi.d -1` 的正确性**：`lddir` 产出的结果格式是 `{PPN, V=1}`，即有效物理地址 OR 1。`addi.d -1` 的作用是清除 V 标志（bit 0=1 → 0），使地址从 `PA|1` 变为 `PA`。由于页表物理地址总是页对齐（低 12 位为 0），bit 0 = 0，减 1 不会借位到高位。**这是正确的用法**。

#### 问题 10：MAXVA 语义兼容 — 最终阻塞 ⭐⭐⭐⭐⭐

**现象**：`usertests -q` 中 `MAXVAplus` 测试失败。用户态对高地址的写入没有稳定失败，某些地址经 LoongArch TLB lookup 或软件 refill 后可能别名到低地址页表项。

**根因**：LoongArch QEMU `la464` 的有效虚拟地址位数（VALEN=48）远超 xv6 原始 Sv39 的 39 位。之前把 `MAXVA` 放宽到 `1<<46` 破坏了 xv6 的用户地址契约：
- `usertests` 认为 `1<<38` 以上的地址应失败
- 高 VA 在 LoongArch TLB lookup 中可能被截断，导致 `MAXVAplus` 构造的非法地址没有按 xv6 语义失败

**解决方案**：

1. **恢复 MAXVA = 1<<38**（与 xv6 原始 Sv39 一致）
2. **TLB refill 入口边界检查**：在 `lddir` 前拒绝 `VA >= MAXVA` 和 `VA < 2*PGSIZE`
3. **低页 guard**：LoongArch `exec` 预映射低 2 页为非用户 guard（`PTE_U=0`），用户程序从 `0x2000` 链接运行
4. **VM 路径检查**：`vmfault()` 拒绝 `va >= MAXVA` 和 `va < 2*PGSIZE`
5. **RVACFG 硬件辅助**：`kvminithart()` 写 `RVACFG=8` 降低硬件有效 VA

## 五、关键经验总结

1. **TLB refill handler 是 LoongArch 移植的核心难点**。必须在 DA=1 约束下正确 walk 多级页表，且 walk 的级数必须与 PWCL/PWCH 配置一致。

2. **lddir/ldpte 产出的 TLBELO 条目中，内存 PTE 的 bit7(P) 和 bit8(W) 是保留位**，必须用 `bstrins.d` 或移位操作清除，否则 TLB 条目被污染。

3. **MAXVA 不能随意放宽**。LoongArch 硬件支持更宽 VA，但必须通过 C 路径、TLB refill 边界检查、RVACFG 三层防护保证 xv6 原始地址契约。

4. **`-bios` 加载方式是裸机启动**，需要自己处理 flash/RAM 分离（`.data` 需要从 flash 搬运到 RAM）。
